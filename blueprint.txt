# SvxLink Node Builder — Engineering Ready Blueprint

## Purpose

This document defines the information required before implementation of the next-generation SvxLink Node Builder architecture.

The objective is to move from:

* line-editing and patching svxlink.conf
* hardware-specific special cases
* duplicated dashboard logic

To:

* a model-driven node definition framework
* deterministic rendering
* feature composition
* guided appliance-style deployment

---

# 1. Core Project Definition

## Required From You

Please define:

### A. Supported Initial Platforms

For each platform:

* platform name
* GPIO assumptions
* audio assumptions
* mandatory hardware
* optional hardware
* known unsupported configurations

Example:

```text
NanoPi-Neo
- Fixed GPIO
- CM108 GPIO unsupported
- USB audio expected
- Single radio only
```

---

### B. Initial Node Types

Define:

```text
Simplex
Repeater
```

For each:

* required sections
* optional sections
* mandatory behaviours
* prohibited behaviours

Example:

```text
Repeater
- Roger mandatory
- Tail handling required
- Duplex audio assumptions
```

# 2. Configuration Model

## Required From You

Define the authoritative node model.

This is the most important design step.

Example structure:

```python
node = {

    'node_type': 'simplex',
    'callsign': 'G7ABC',
    'language': 'en_US',
    'reflector': {},
    'ident': {},
    'roger': {},
    'modules': []
}
```

For each key:

* possible values
* defaults
* validation rules
* dependencies

# 3. Rendering System

## Required From You

Define:

### A. Canonical Source Strategy

Choose one:

```text
1. Full text templates with placeholders
2. Python-generated sections
3. Hybrid approach
```

Current recommendation:

```text
Hybrid approach
```

Meaning:

* stable boilerplate sections remain templated
* dynamic feature blocks rendered in Python

---

### B. Placeholder Convention

Example:

```ini
LOGICS={{LOGICS}}
{{LINKS_BLOCK}}
```

Please define preferred syntax.

---

### C. Section Ordering Rules

Define canonical section order.

Example:

```text
GLOBAL
Rx1
Tx1
SimplexLogic
RepeaterLogic
Modules
ReflectorLogic
Macros
```

This becomes deterministic renderer behaviour.

# 13. Most Important Principle

The user must never directly manipulate:

* svxlink.conf
* Tcl logic
* GPIO mappings
* raw module declarations

Instead:

The user describes the radio node they want.

The framework constructs a valid operational system.

/start
/platform
/node
/interface
/squelch
/ident
/modules
/reflector
